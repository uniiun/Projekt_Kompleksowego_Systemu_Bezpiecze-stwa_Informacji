1. Cel i opis projektu
Celem projektu jest zaprojektowanie i wykonanie aplikacji webowej, która kontroluje dostęp użytkowników do zasobów informacyjnych organizacji. System ma umożliwiać przechowywanie dokumentów, przypisywanie ich do działów, oznaczanie poziomu poufności oraz sprawdzanie, czy dany użytkownik może wykonać konkretną operację, np. odczyt, pobranie, dodanie, edycję lub usunięcie dokumentu.
Projekt realizuje temat: „Projekt systemu kontroli dostępu do zasobów informacyjnych z wykorzystaniem wybranej technologii”. Wybraną technologią jest Python z frameworkiem Django oraz Django REST Framework. System może być zrealizowany jako aplikacja z osobnym frontendem React albo jako prostsza aplikacja Django z szablonami HTML. Dla pracy zespołowej zalecany jest wariant Django REST API + React.
System pokazuje praktyczne zastosowanie kontroli dostępu do informacji.
Użytkownicy mają różne role oraz różne poziomy uprawnień.
Zasoby są przypisane do działów oraz poziomów poufności.
Każda istotna operacja jest zapisywana w dzienniku audytu.
Administrator ma możliwość zarządzania użytkownikami, dokumentami i logami.
2. Kontekst biznesowy
Przykładowym odbiorcą systemu jest firma lub instytucja posiadająca kilka działów organizacyjnych. Każdy dział wytwarza i przechowuje dokumenty, które nie zawsze powinny być widoczne dla wszystkich pracowników. System ma ograniczać ryzyko nieuprawnionego dostępu do informacji.
Dział
Przykładowe zasoby
IT
Procedury bezpieczeństwa, dokumentacja techniczna, instrukcje konfiguracji
HR
Regulaminy, dane pracownicze, dokumenty kadrowe
Finanse
Raporty finansowe, zestawienia kosztów, budżety
Zarząd
Strategie, decyzje zarządcze, dokumenty tajne
Sprzedaż
Oferty, raporty sprzedaży, bazy klientów testowe
Problem biznesowy polega na tym, że użytkownicy organizacji powinni widzieć tylko te dane, które są im potrzebne do pracy. Pracownik działu HR nie powinien widzieć raportów finansowych, zwykły pracownik nie powinien usuwać dokumentów, a audytor powinien mieć dostęp głównie do odczytu i historii operacji.

Role są podstawą mechanizmu RBAC, czyli Role-Based Access Control. W systemie można dodatkowo sprawdzać dział użytkownika oraz poziom poufności dokumentu. Dzięki temu mechanizm nie jest tylko prostym podziałem na role, ale uwzględnia też kontekst biznesowy.
4. Zasoby informacyjne
Głównym zasobem informacyjnym w systemie jest dokument. Dokument może reprezentować plik PDF, plik tekstowy, raport, procedurę lub wpis informacyjny. Każdy dokument ma właściciela, dział, poziom poufności oraz historię dostępu.

Strategia zarządu
Każdy dokument ma tytuł, opis, plik, dział, poziom poufności, autora i daty utworzenia oraz modyfikacji.
Dokument może mieć przypisanych użytkowników z indywidualnym dostępem.
System nie pokazuje dokumentów, do których użytkownik nie ma prawa dostępu.
Próba wejścia bezpośrednim linkiem do zakazanego dokumentu kończy się odmową dostępu.

6. Logika systemowa
Logika systemowa opisuje, jak aplikacja podejmuje decyzję o przyznaniu albo odmowie dostępu. Najważniejsze jest to, że frontend nie może sam decydować o bezpieczeństwie. Frontend może ukrywać przyciski, ale ostateczna kontrola zawsze musi znajdować się po stronie backendu.
6.1. Algorytm decyzji dostępowej
Sprawdź, czy użytkownik jest zalogowany.
Pobierz rolę użytkownika oraz przypisany dział.
Pobierz metadane dokumentu: dział, poziom poufności, autora i listę użytkowników z indywidualnym dostępem.
Jeżeli użytkownik jest administratorem, przyznaj dostęp.
Jeżeli dokument ma poziom PUBLIC, przyznaj dostęp do odczytu zalogowanym użytkownikom.
Jeżeli dokument należy do działu użytkownika i poziom poufności pozwala na dostęp dla jego roli, przyznaj dostęp.
Jeżeli użytkownik znajduje się na liście indywidualnego dostępu, przyznaj dostęp.
W przeciwnym wypadku odmów dostępu, zwróć błąd 403 i zapisz zdarzenie w logach.
6.2. Pseudokod kontroli dostępu
def can_access_document(user, document, action):    if not user.is_authenticated:        return False    if user.role == "ADMIN":        return True    if action in ["DELETE", "MANAGE_USERS"]:        return False    if document.confidentiality_level == "PUBLIC" and action == "VIEW":        return True    if user in document.allowed_users.all():        return True    if user.role == "MANAGER" and user.department == document.department:        return action in ["VIEW", "DOWNLOAD", "CREATE", "UPDATE"]    if user.role == "EMPLOYEE" and user.department == document.department:        return document.confidentiality_level in ["PUBLIC", "INTERNAL"] and action in ["VIEW", "DOWNLOAD"]    if user.role == "AUDITOR":        return action in ["VIEW", "DOWNLOAD", "VIEW_LOGS"]    return False
7. Przepływy działania systemu
7.1. Logowanie użytkownika
Użytkownik otwiera aplikację i widzi formularz logowania.
Wpisuje e-mail oraz hasło.
Frontend wysyła dane do endpointu logowania backendu.
Backend sprawdza poprawność danych.
Po poprawnym logowaniu backend zwraca token JWT lub tworzy sesję.
Frontend zapisuje token i przekierowuje użytkownika do panelu głównego.
System pokazuje funkcje zgodne z rolą użytkownika.
7.2. Wyświetlenie listy dokumentów
Użytkownik przechodzi do zakładki Dokumenty.
Frontend wysyła zapytanie GET /api/documents/.
Backend rozpoznaje użytkownika na podstawie tokena.
Backend filtruje dokumenty według roli, działu i poziomu poufności.
Frontend otrzymuje tylko dokumenty, które użytkownik ma prawo widzieć.
Operacja odczytu listy może zostać zapisana w logach jako VIEW_LIST.
7.3. Próba nieautoryzowanego dostępu
Użytkownik próbuje wejść w adres szczegółów dokumentu, którego nie powinien widzieć.
Frontend wysyła zapytanie GET /api/documents/{id}/.
Backend wykonuje pełną kontrolę dostępu.
Jeżeli użytkownik nie spełnia reguł, backend zwraca 403 Forbidden.
System zapisuje w AccessLog zdarzenie ACCESS_DENIED.
Frontend pokazuje komunikat: Brak uprawnień do tego zasobu.
7.4. Dodanie dokumentu przez menedżera
Menedżer wybiera opcję Dodaj dokument.
Wypełnia tytuł, opis, dział, poziom poufności i opcjonalnie dodaje plik.
Backend sprawdza, czy menedżer może dodawać dokument do wskazanego działu.
Jeśli tak, dokument zostaje zapisany w bazie.
System zapisuje log CREATE_DOCUMENT.
Nowy dokument pojawia się na liście uprawnionych użytkowników.

Nie commitować bezpośrednio do main.
Każde zadanie wykonywać na osobnej gałęzi feature.
Po zakończeniu zadania tworzyć Pull Request do develop.
Nie wrzucać folderów venv, node_modules, plików .env ani haseł.
Osoba odpowiedzialna za dokumentację pilnuje zgodności API z frontendem.

15. Podsumowanie
Projekt powinien zostać zrealizowany jako system kontroli dostępu do dokumentów organizacji. Najważniejszym elementem nie jest samo przechowywanie plików, ale mechanizm decyzyjny sprawdzający, czy użytkownik może zobaczyć lub zmodyfikować konkretny zasób. Dzięki zastosowaniu ról, działów, poziomów poufności i logów audytu projekt dobrze odpowiada na temat systemu kontroli dostępu do zasobów informacyjnych.
